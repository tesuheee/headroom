Headroom Architecture Guide

Claude と Codex の使用量ページを WebView2 で読み取り、残量・使用量・リセット時刻を 小さなデスクトップウィジェットとして表示するアプリの設計資料です。 実装、状態、UI、リリースの関係をブラウザで追えるようにまとめています。 このHTML単体でも、アプリの目的、構成、状態管理、描画、設定、ビルド、配布方針を説明できる密度を持たせています。

Architecture

実装は小さくまとまっていますが、読み方としては UI、アプリ制御、ドメイン状態、 WebView2・ローカルファイルの4層で見ると分かりやすいです。

UI

UsageForm

メインウィジェット。カード、バー、サイドレール、クリック判定、設定画面呼び出しを担当します。

State

ServiceState / UsageData

Claude / Codexごとの現在状態、取得データ、上限到達、表示用アニメーション値を保持します。

Infra

WebView2

ログイン済みページを実際に開き、レンダリング済みテキストを取得してパースします。

設計方針

Headroom は、Claude と Codex が公式に公開している利用状況ページを、ユーザー自身のログインセッションで表示し、 そこに出ているテキストから利用枠の状態を読み取ります。外部APIキーや独自サーバーを使わないため、 アプリは「ローカルでWebページを読むウィジェット」として閉じています。

UIは小さな常駐ウィジェットであることを優先し、標準的なフォーム部品を並べるのではなく、 透明フォームにカード・バー・アイコンを直接描画します。これによりサイズ、色、余白、バー位置を細かく制御できます。 一方で、標準コントロールよりもレイアウト崩れやヒット判定の責務がアプリ側に寄るため、 DrawService、DrawRow、DrawBar、hits 辞書の整合性が重要になります。

設計判断 理由 注意点
WebView2でログイン済みページを読む サービス側の公式画面に表示される値をそのまま利用でき、APIキー管理が不要になるため。 ページ文言やDOM構造の変更に弱い。デバッグ用テキスト保存で調査できるようにする。
サービス別パーサを持つ Claude と Codex ではページ構成、単位、使用量/残量の出方が違うため。 共通化しすぎると例外処理が読みにくくなるため、抽出ロジックは分けて読む。
使用量と残量を相互変換する ユーザーが Rem / Used を切り替えられるようにするため。 色判定は常に残量基準に戻してから行う。使用量表示時に黄色判定が逆転しないよう注意する。
設定変更はライブプレビューする レイアウトや表示モードの違いを即座に確認できるようにするため。 キャンセル時に元の設定へ戻すため、設定画面を開いた時点のコピーが必要。

Runtime Flows

起動後は設定を読み、WebView2を初期化し、定期更新または手動更新で Claude / Codex のページ本文を取得します。

1. LaunchWidgetSettings.Load() で設定を復元
2. WebView2Claude / Codex 用の非表示WebViewを初期化
3. Navigate使用量ページへ移動し本文が安定するまで待つ
4. Parseサービス別パーサで使用量・残量・リセットを抽出
5. StateServiceState と表示用アニメーション値を更新
6. PaintOnPaint でカードとバーを描画

起動時に行う処理

起動時には、まずローカル設定を読み込み、ウィンドウサイズ、表示サービス、言語、表示モード、更新間隔、しきい値を復元します。 次に WebView2 のユーザーデータフォルダを確認し、旧ビルドの AiUsageWebView2 プロファイルしか存在しない場合は Headroom のプロファイルへコピーします。これにより、アプリ名変更後もログインセッションを引き継げます。

WebView2 の初期化が終わると、Claude と Codex の初回更新を走らせます。ログイン済みであれば利用状況を取得し、 未ログインであればカードにログインが必要な状態を表示します。ログイン状態そのものはアプリが直接保持するのではなく、 WebView2プロファイル内のCookie・セッションに委ねます。

フロー 主な責務 失敗時の扱い
起動 設定ロード、WebView2プロファイル確認、フォーム初期化、初回更新。 設定が読めない場合はデフォルト設定で起動する。
定期更新 表示中サービスの更新期限を判定し、必要なサービスだけWebView2で再取得する。 多重更新は IsRefreshing で防ぐ。取得失敗時はエラー状態を表示する。
手動更新 カード右上のRefresh操作で対象サービスだけ即時更新する。 通常更新と同じパース・状態更新経路を使う。
Boost 対象サービスの更新間隔を一時的に短くする。 期限切れ、または再クリックで通常更新へ戻る。
ログイン ログイン用フォームで対象サービスのページを開き、ユーザー操作で認証する。 フォームを閉じたあと、対象サービスを再更新する。
ログアウト 対象サービスのCookieを削除し、WebViewを空ページへ移動する。 状態をログイン必要へ戻す。

Data & State

Claudeは主に使用済み、Codexは主に残量としてページに表示されるため、アプリ内では UsageData が両方の表示モードに変換できる形で値を持ちます。

元データ 残量表示 使用量表示 色判定
Remaining = 75 75% 25% 残量75%として判定
Used = 40 60% 40% 残量60%として判定
Remaining = 0 または limit reached 0% 100% 上限到達状態として表示

データモデルの役割

ServiceState は、Claude と Codex それぞれの「現在動いている状態」を持ちます。 ここには、取得済みの UsageData、更新中かどうか、最後に更新した時刻、Boost期限、バーアニメーション用の表示値が入ります。 永続化される設定ではなく、アプリが起動している間だけ意味を持つランタイム状態です。

UsageData は、ページ本文から抽出した利用枠データです。Claude は使用済み%を中心に取れる一方で、 Codex は残量%として読めることが多いため、使用済みと残量のどちらも表現できる形にしています。 表示時には、設定に応じて残量表示または使用量表示へ変換します。

WidgetSettings は、ウィンドウサイズ、言語、表示するサービス、横/縦レイアウト、サービス別の Rem / Used、 リセット表示形式、通常更新間隔、Boost更新間隔、黄色・赤のしきい値を持ちます。 保存先は Application.StartupPath/settings.json で、通常は bin/settings.json です。

パース方針

サービス 5時間枠 週間枠 特徴
Claude Current session / 現在のセッション を中心に読む。 Weekly limit / 週間制限 / All models を中心に読む。 主に使用済み%として抽出し、残量表示時は 100 - used で変換する。
Codex 5 hour / 5時間の使用制限 を中心に読む。 weekly / 週あたりの使用制限 を中心に読む。 主に残量%として抽出する。credits セクションは利用枠と別扱いにする。

上限到達と色判定

上限到達は、数値として残量が0%以下になった場合と、ページ本文に limit reached 系の文言が出ている場合の両方で判断します。 カード全体の上限到達表示と、5時間枠・週間枠それぞれのバー色は別の責務です。 例えば5時間枠だけ上限に到達していても、週間枠に十分な残量があれば、週間枠は通常色で表示できます。

バー色は「現在の表示が使用量か残量か」に関係なく、最終的に残量へ戻してから判定します。 この方針により、使用量表示に切り替えても「残りが少ないほど黄色・赤」という意味が保たれます。

UIレイアウト仕様

メインウィジェットは、表示するサービス数とレイアウト設定に応じて理想サイズを変えます。 Claude と Codex の両方を表示する場合は横並びまたは縦並びを選べ、片方だけ表示する場合は単一カードのコンパクトなサイズになります。 右端にはサイドレールがあり、アプリ終了、最前面固定、残量/使用量切替、リセット表示形式切替、設定画面を操作できます。

表示状態 標準サイズ 用途
両サービス・横並び 760 x 154 デスクトップ上部や下部に横長で置く用途。
両サービス・縦並び 390 x 318 画面端に縦長で置く用途。
片サービスのみ 390 x 154 Claudeだけ、またはCodexだけを監視する用途。

カード仕様

各サービスカードは、ヘッダー、操作ボタン、5時間枠、週間枠で構成されます。 ヘッダーにはサービス色のドット、サービス名、上限到達時のバッジ、Boostボタン、Refreshボタンを表示します。 行には枠名、Rem/Used、パーセント値、バー、リセット時刻を表示します。

パーセント値は桁数が変わってもバーの開始位置が動かないよう、固定幅の列として扱います。 1桁パーセント、2桁パーセント、100%のいずれでもバー位置が揺れないことが重要です。 英語UIではカード内の横幅を節約するため Remaining ではなく Rem を使い、 設定画面や説明文では省略せず Remaining を使います。

Releaseにはバージョン付きの Headroom-vX.Y.Z.zip だけを添付します。 バージョンなしzipは、同じRelease内で重複して見え、利用者が迷いやすいため置きません。

成果物の方針

リリースzipには、実行に必要な最小限のファイルだけを入れます。 settings.json、証明書関連ファイル、デバッグ出力、旧exe、一時ファイル、NuGet展開物全体は含めません。 これにより、ユーザー固有の設定や認証情報が誤って配布物に混ざることを避けます。

含める 理由
Headroom.exeアプリ本体。
Microsoft.Web.WebView2.WinForms.dllWebView2 WinForms統合。
Microsoft.Web.WebView2.Core.dllWebView2 Core API。
WebView2Loader.dllWebView2 Runtimeの読み込み。

今後の改善候補

現在の構成は小規模アプリとしては十分扱いやすい一方で、機能が増える場合は単一の Program.cs が大きくなりすぎます。 今後の主な改善候補は、機能別ファイル分割、GitHub Actionsによるビルド・リリース自動化、 バージョン番号のソースコード管理、READMEスクリーンショット生成手順のスクリプト化です。

Design Specification

この章は、納品用の設計説明として読めるように、システムの目的、範囲、責務分担、データライフサイクル、 セキュリティ、運用上の注意点を文章で整理したものです。

1. システム目的

Headroom は、Claude と Codex の利用枠を確認するために毎回ブラウザで設定画面を開く手間を減らすことを目的とした Windows向けデスクトップウィジェットです。ユーザーのローカル環境でサービスの使用量ページを開き、 画面に表示されている利用状況を読み取り、5時間枠と週間枠の状態をコンパクトに表示します。

本アプリは使用量を管理・制御するものではなく、既存サービスの画面に出ている情報を見やすく表示する補助ツールです。 使用量の計算元、上限、リセット時刻の最終的な正しさは Claude / Codex 側の表示に依存します。

2. スコープ

対象 内容
対象サービス Claude と Codex の使用量ページ。
対象プラットフォーム Windows。Windows Forms と WebView2 Runtime が前提。
表示対象 5時間枠、週間枠、使用量/残量、リセット時刻、上限到達状態。
操作対象 手動更新、Boost、ログイン、ログアウト、表示設定、しきい値設定。
対象外 使用量の購入、プラン変更、サービス側の上限変更、独自サーバーでの同期。

3. コンポーネント責務

コンポーネント 責務 変更時に注意する点
UsageForm メイン画面、WebView2管理、更新、描画、クリック処理、設定画面呼び出し。 責務が広いため、描画変更と更新処理変更を混ぜない。ヒット判定の矩形も合わせて確認する。
SettingsForm 設定項目の編集、ライブプレビュー、ログイン・ログアウト操作。 キャンセル時に元設定へ戻るか、OK時だけ保存されるかを確認する。
ServiceState サービスごとのランタイム状態を保持する。 永続設定と混同しない。起動中だけ意味がある値を置く。
UsageData パース後の使用量・残量・リセット・上限到達情報を保持する。 Claude/Codexで元データの意味が違うため、表示前の変換を慎重に扱う。
WidgetSettings ユーザー設定の読み書きとデフォルト値を管理する。 新しい設定を追加する場合は、デフォルト値、保存、読み込み、設定画面、プレビューを揃える。
build.ps1 WebView2依存DLLの準備、コンパイル、署名、リリースzip作成。 リリースzipにローカル設定や不要ファイルを混ぜない。

4. データライフサイクル

利用枠データは、サービスページから取得したテキストを起点にして、パース、正規化、表示モード変換、色判定、描画という順番で流れます。 この流れの途中で永続化されるのはユーザー設定だけで、取得した使用量データそのものは主に画面表示のためのランタイム状態です。

5. 永続化とローカルファイル

場所 内容 扱い
bin/settings.json ウィンドウサイズ、言語、表示サービス、更新間隔、しきい値など。 ユーザー環境ごとの設定。Gitにもリリースzipにも含めない。
%LOCALAPPDATA%/Headroom/WebView2Profile WebView2のCookie、ログインセッション、ブラウザプロファイル。 ローカル環境に保存。アプリは認証情報を独自送信しない。
%USERPROFILE%/.headroom 取得したページ本文やエラー調査用テキスト。 パース不具合調査用。公開や配布には含めない。
releases/ リリースzip成果物。 Git管理外。GitHub Releaseへ必要なzipだけ添付する。

6. セキュリティと認証情報

Headroom は、Claude / Codex のログインIDやパスワードをアプリ固有の設定ファイルに保存しません。 ログインはWebView2内でユーザーが通常のWebログインとして行い、その結果のCookieやセッションはWebView2プロファイルに保存されます。 アプリはそのログイン済みページを表示・読み取りするだけです。

リリース時には、settings.json、WebView2プロファイル、証明書、デバッグ出力、スクレイピング結果を配布物に含めないことが重要です。 これらはユーザー環境固有の情報、または調査用のローカル情報であり、公開リポジトリにもRelease assetにも入れません。

7. エラー処理と保守

最も起こりやすい障害は、Claude または Codex 側の画面文言変更によるパース失敗です。 そのため、取得した本文をデバッグファイルとして保存し、どの文言が変わったのかを後から確認できるようにしています。 取得失敗時はアプリ全体を終了するのではなく、該当カードをエラーまたはログイン必要状態にして、再試行できる状態を保ちます。

リスク 影響 対策
サービス側の文言変更 使用量やリセット時刻を抽出できなくなる。 デバッグ本文を確認し、サービス別パーサを更新する。
ログインセッション切れ 利用状況ページへアクセスできなくなる。 ログイン必要状態を表示し、ログイン画面へ誘導する。
UIの桁ずれ バー位置やテキスト位置が揺れる。 固定幅列、右揃え、表示パターン別スクリーンショットで確認する。
不要ファイルの配布混入 ローカル設定や不要DLLがユーザーへ配布される。 build.ps1 でzip対象を明示列挙し、.gitignore を維持する。

8. 変更時の確認観点

表示やパースを変更した場合は、Claudeのみ、Codexのみ、両サービス横並び、両サービス縦並び、残量表示、使用量表示、 1桁パーセント、100%、上限到達、ログイン必要状態を確認します。 特に、パーセント値の桁数が変わったときにバーの開始位置が動かないこと、英語と日本語でラベルがはみ出さないこと、 残量基準の色判定が表示モードに依存して壊れないことが重要です。
